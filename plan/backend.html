План: Backend

API доставки фич и дельт; CI/CD-воркер для построения манифестов и версий; сравнение версии клиента с накатанной, формирование и кеширование дельт.

Контекст

  • Фича — небольшой проект со своим манифестом и набором файлов-узлов. Узел: id (UUID от пути), name (имя файла), hash (content-hash). Иерархия задаётся манифестом (дерево папок и файлов).
  • Дельта строится сервером при сравнении переданной клиентом версии и версии, опубликованной через админку; хранится только в кешах сервера.
  • Клиент передаёт версию и идентификатор фичи; сервер достаёт манифесты двух версий, сравнивает, строит дельту (или достаёт из кеша) и отправляет клиенту дельту вместе с опубликованным манифестом (либо «изменений не было»).

Задачи

  1. CI/CD-воркер — периодический анализ ветки проекта; построение дерева хешей и манифестов по фичам (ios/android/web); версионирование; сохранение артефактов в хранилище (.git или подпапка backend).
  2. API фич и дельт — эндпоинт запроса фичи (платформа, id фичи, версия). Холодный старт и проверка обновлений: сервер отдаёт дельту вместе с опубликованным манифестом. Если версия совпадает — «изменений не было».
  3. Хранение и структура — манифесты и файлы фич; версия «опубликованная» для каждой платформы в конфиге или БД.
  4. Обратная совместимость — сохранить GET /api/v1/files, GET /api/v1/files/{platform}, GET /api/v1/files/{platform}/{filename} для статичных файлов.
Результат: Клиенты всегда получают по фиче дельту вместе с опубликованным манифестом (при отличии версий) или «изменений не было»; дельты кешируются на сервере.
